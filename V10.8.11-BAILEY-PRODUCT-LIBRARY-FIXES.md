## Finding 1 — Product page "isn't loading" is NOT a hang. It's a subscriber false-negative.

**What the white pills are**: There is zero skeleton/`animate-pulse` code in the
`/products/[id]` render path. The pills are the **5 `GatedTeaser` blur cards** in
`ProductIntelligenceSection.tsx:83-109` — the anonymous "Subscribe to unlock" teasers,
each with a `backdrop-blur-[2px]` overlay and a grey pill button. To a paying subscriber
they read as broken/stuck placeholders.

**Why a subscriber sees the anonymous gated view** (root cause):
`ProductIntelligenceSection` gates on a client-side `supabase.auth.getSession()` call in a
`useEffect` (line 37). On the **public** `/products/[id]` route (outside the authenticated
`(app)` AppShell), the Supabase client has often not rehydrated the session from
localStorage by the time that one-shot `getSession()` fires → it returns `null` →
`setIsSubscriber(false)` at line 39 → the endpoint `/api/me/subscription` is never even
called → anonymous GatedTeasers render for a paying user.

This is the **v10.7.0 Phase F bug reproducing**. Phase F moved the *subscription check*
to a server endpoint (correct), but left the *session detection* as a one-shot
`getSession()` (still racy). The endpoint is sound; the gate in front of it is not.

**DB confirmation** (`scripts/check-bailey-sub.ts`): Bailey's
`ss_user_profiles.plan = 'pro_monthly'`, `ss_subscriptions = null` (she was manually set
to pro in v5.5.0, never went through Stripe). `hasActiveSubscription()`
(`src/lib/subscription.ts:33-54`) has a profile-plan fallback, so the server endpoint
returns `{ active: true }` for her **if it's ever called**. The bug is purely that the
client never calls it because `getSession()` raced to null.

**Fix**: Replace the one-shot `getSession()` in `ProductIntelligenceSection` with the
app-wide `useAuth()` context (`src/contexts/AuthContext.tsx`), which listens to
`onAuthStateChange` and therefore catches the session once it hydrates. Gate the
subscription fetch on `useAuth().loading === false` so we only decide subscriber state
after auth has settled. While `loading`, render `null` (no flash) — same as today's
`isSubscriber === null` branch.

This eliminates the race for every visitor on the public route, not just Bailey.

## Finding 3 — Image coverage: is "eventually all shown" accurate?

**Audited answer: partially accurate, with important caveats.**

- **93.0% of products (5,507 / 5,924)** already have a non-null `image_url`. The 417
  image-less are almost entirely the Feb 2026 seed cohort (415 of 417). Products created
  since March 2026 are ~100% covered.
- **New products auto-capture images**: the daily `scan-korean-products` cron extracts
  `image_url` from Olive Young listings (`src/lib/pipeline/sources/olive-young.ts:189-199`
  → `extractor.ts:167` → `batch-processor.ts:237`). So going forward, new catalog products
  get photos automatically.
- **There is NO active backfill for existing image-less products.** Nothing re-scrapes the
  ~415 legacy products. The only "retry" cron (`retry-enrichment-queue`, monthly) is
  ingredient-only and currently a no-op placeholder.
- **Custom routine steps will NEVER have a product image, by design**: "Cool water rinse,"
  "Ice roller / cold spoon," "LED mask," "Shower / cleanse," and custom-entry products
  (NULL `product_id`) have no catalog record to attach an image to. Bailey's routine has 8
  such steps.
- **~3% of products have an image_url that points to a brand-direct/Shopify URL** which can
  go dead (404/drift), so a small minority render blank despite having a URL.

**Honest version of the answer to Bailey**:
> "Most products (93%) already have photos, and new ones get photos automatically as
> they're added. A batch of ~415 older products are missing photos and I need to run a
> one-time backfill to fill those in — that's on the list. And the device/manual steps
> (ice roller, LED mask, water rinse) won't ever show a product photo because they aren't
> catalog products."

**Decision (Scott, May 26)**: Do NOT build the backfill this session. Documented as a
deferred task below.

### Deferred task — Legacy product image backfill (~415 products)
- Write `scripts/backfill-product-images.ts`: for each verified product with
  `image_url IS NULL`, search Olive Young (and fall back to other sources) for the product
  by brand+name, extract the listing thumbnail, update `image_url`. Confidence-gate the
  match (don't attach the wrong product's photo).
- Optionally add a monthly cron that re-runs it so the catalog self-heals.
- Estimated: a few hours. Trigger: when Scott wants the "all photos eventually" promise to
  be literally true, or when image-less products become a visible share of what Bailey/users
  browse.
